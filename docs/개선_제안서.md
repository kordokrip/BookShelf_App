## 1. 보안 (Security)

### 🔴 HIGH — `optionalAuth` demo-user 폴백

**현재 상태**: `worker/auth.ts`의 `optionalAuth` 미들웨어에서 JWT 토큰이 없으면 `userId = 'demo-user'`로 폴백합니다.

```typescript
// worker/auth.ts optionalAuth
토큰 없음 → userId = 'demo-user'  // ⚠️ 인증 없이 데이터 접근 가능
```

**문제점**:
- GET 라우트에서 인증 없이 'demo-user'의 책/노트 데이터가 반환됩니다.
- 실제 demo-user로 데이터가 쌓이면 의도치 않은 정보가 노출될 수 있습니다.

**제안**: 토큰 없으면 빈 배열 반환 또는 401 응답을 명시. demo-user 폴백 완전 제거.

---

### 🔴 HIGH — CORS 와일드카드 범위 과도

**현재 상태**:
```typescript
allowOrigin: ['http://localhost:*', 'https://bookshelf*.pages.dev', 'https://bookshelf*.workers.dev']
```

**문제점**: `bookshelf*` 패턴이 지나치게 광범위하여 `bookshelf-evil.workers.dev` 같은 제3자 도메인도 허용됩니다.

**제안**: 정확한 도메인만 허용하도록 변경:
```typescript
allowOrigin: [
  'http://localhost:5173',
  'https://bookshelf-api.kordokrip.workers.dev',
  // 필요한 정확한 도메인만 추가
]
```

---

### 🟡 MEDIUM — JWT Refresh Token 미사용

**현재 상태**: `wrangler.toml`에 `SESSIONS` KV 바인딩이 선언되어 있으나, 실제 Refresh Token 로직이 미구현. JWT 24시간 만료 후 재로그인 필요.

**제안**: SESSIONS KV를 활용한 Refresh Token (30일 TTL) 구현:
- `POST /api/auth/refresh` → 새 Access Token 발급
- 기존 SESSIONS KV 바인딩 활용

---

### 🟡 MEDIUM — PATCH /api/users/profile zod 검증 미적용

**현재 상태**: `routes/users.ts`의 PATCH 핸들러에서 `zValidator` 없이 `c.req.json()` 직접 사용, 각 필드 수동 검증.

**제안**: 다른 라우트와 동일하게 `zValidator('json', updateProfileSchema)` 적용.

## 2. 백엔드 리팩토링

### 🟡 MEDIUM — `cover_image` 필드 이중 의미

**현재 상태**: `books.cover_image`에 R2 키(`covers/userId/bookId.jpg`)와 외부 URL(`https://...`)이 혼재.

```typescript
// GET /api/books/:id/cover
if (r2Key.startsWith('http')) → redirect
else → R2.get(r2Key)
```

**제안**:
- `cover_image_source` 컬럼 추가 (`'r2' | 'external'`)
- 또는 R2 키에 prefix 규약 적용 (`r2://covers/...`)

---

### 🟡 MEDIUM — reading_sessions 중복 방지 없음

**현재 상태**: `reading_sessions` 테이블에 UNIQUE 제약이 없어, 네트워크 재시도/더블탭 시 동일 세션이 중복 삽입될 수 있습니다.

**제안**:
```sql
CREATE UNIQUE INDEX idx_sessions_unique ON reading_sessions(book_id, session_date, pages_read);
```
또는 클라이언트에서 idempotency key를 전송.

---

### 🟢 LOW — is_overdue 필드 수동 관리

**현재 상태**: `books.is_overdue`는 API 호출 시에만 갱신되며, `goal_date` < 현재 날짜인데도 자동 반영되지 않습니다.

**제안**: Worker 레벨에서 Generated Column 또는 조회 시 계산:
```sql
-- 조회 시 동적 계산 (D1/SQLite 호환)
SELECT *, CASE WHEN goal_date < date('now') AND status='reading' THEN 1 ELSE 0 END AS is_overdue ...
```

## 3. 프론트엔드 리팩토링

### 🟡 MEDIUM — `checkAuth()` 레이스 컨디션

**현재 상태**: `App.tsx`에서 Google OAuth 콜백 시:
```typescript
useEffect(() => {
  const token = searchParams.get('token')
  if (token) localStorage.setItem('auth_token', token)
  checkAuth()
}, [])
```
`history.replaceState`로 URL 클린업 전 리렌더링 시 `searchParams.get('token')`이 다시 호출될 수 있습니다.

**제안**: `useRef`로 초기화 1회 보장 또는 URL 클린업 후 checkAuth 호출.

---

### 🟡 MEDIUM — BottomNavBar 이중 쿼리

**현재 상태**: Root 레이아웃에 마운트되므로 모든 보호 라우트에서 항상 실행:
```typescript
const readingBooks = useBooks({ status: 'reading' })  // 쿼리 1
const wishBooks    = useBooks({ status: 'wish' })      // 쿼리 2
```

**제안**: `select` 옵션으로 count만 구독 (현재 이미 부분 적용):
```typescript
const readingCount = useBooks({ status: 'reading' }, { select: d => d.count })
```
또는 `/api/books/counts` 전용 경량 엔드포인트 도입.

---

### 🟢 LOW — framer-motion 메인 번들 포함

**현재 상태**: `framer-motion` (현재 `motion`)이 vendor-react 청크(285KB)에 포함. tree-shaking 불완전.

**제안**: `vite.config.ts` manualChunks에 별도 청크로 분리:
```typescript
'vendor-motion': ['motion']
```

---

### 🟢 LOW — refetchOnWindowFocus 모바일 이슈

**현재 상태**: 모바일에서 앱 이탈/복귀 시 모든 활성 쿼리가 재요청되어 로딩 깜빡임 발생 가능.

**제안**: `staleTime` 60초로 확대 또는 `refetchOnWindowFocus: false` 설정.

## 4. DB 스키마 개선

### 🟡 MEDIUM — notes type에 'review' 추가 후 체크 제약 미갱신

**현재 상태**: `0003_notes_review_type.sql`에서 type 'review'를 허용하도록 테이블을 재생성했으나, 기존 CHECK 제약과 일관성 확인 필요.

**제안**: D1 마이그레이션에서 CHECK 제약 확인:
```sql
CHECK(type IN ('quote', 'memo', 'review'))
```

---

### 🟢 LOW — reading_sessions 무제한 조회

**현재 상태**: `GET /api/sessions`에서 `limit` 파라미터 미전달 시 전체 반환.

**제안**: 서버 측 기본 `limit=100` + 최대 상한 1000 강제.

---

## 5. 성능 최적화

### 🟡 MEDIUM — 앱 최초 진입 시 동시 요청 폭발

**현재 상태**: BottomNavBar(2개) + 페이지 훅(2~4개) → 앱 첫 진입 시 6~8개 동시 요청.

**제안**:
1. `suspense: true` + React Suspense로 워터폴 제어
2. 또는 `/api/initial-data` 일괄 로드 엔드포인트 도입 (서버에서 D1 batch)

---

### 🟢 LOW — PWA precache 39 entries (1944.57 KiB)

**현재 상태**: 빌드 시 precache 크기가 ~2MB. 느린 네트워크 환경에서 Service Worker 설치 시간 증가.

**제안**: 자주 변경되지 않는 정적 자산만 precache, 나머지는 runtime caching으로 전환.
