## 1. 보안 이슈

### 🔴 SEC-01: 그룹 채팅 XSS 방지

**현황**: `group_messages.content`가 사용자 입력 텍스트 그대로 D1에 저장되고, ChatTab에서 `{msg.content}`로 렌더링된다. React의 JSX 자동 이스케이프로 기본 XSS는 방지되지만, 향후 마크다운 렌더링이나 `dangerouslySetInnerHTML` 도입 시 취약해질 수 있다.

**제안**:
- 서버 측에서 메시지 저장 전 HTML 태그 스트립핑 적용 (`content.replace(/<[^>]*>/g, '')`)
- 프론트엔드에서 URL 자동 링크 변환 시 `rel="noopener noreferrer"` + `target="_blank"` 필수
- Content Security Policy (CSP) 헤더 추가 검토

```typescript
// worker/routes/groups.ts — POST /:id/messages
const sanitized = body.content.replace(/<[^>]*>/g, '').trim();
```

---

### 🔴 SEC-02: CORS 와일드카드 서브도메인 범위 축소

**현황**: `allowOrigin`에 `bookshelf*` 패턴이 포함되어 있어 `bookshelf-evil.workers.dev` 같은 도메인도 허용된다.

**제안**:
```typescript
// worker/index.ts — CORS 설정 변경
allowOrigin: [
  'http://localhost:5173',
  'https://bookshelf-app.pages.dev',         // 정확한 도메인
  'https://bookshelf-api.kordokrip.workers.dev',
]
```

---

### 🔴 SEC-03: `optionalAuth` demo-user 폴백 제거

**현황**: 토큰 없이 GET 요청 시 `userId = 'demo-user'`로 폴백되어, 잠재적으로 demo-user 데이터가 누출된다.

**제안**: 토큰 없으면 `userId = null`로 설정하고, 각 라우트에서 null 체크 후 빈 배열 반환.

```typescript
// worker/auth.ts — optionalAuth 수정
if (!token) {
  c.set('userId', null);
  return next();
}
```

---

### 🟡 SEC-04: JWT Refresh Token 미구현

**현황**: SESSIONS KV 바인딩이 선언되어 있으나 Refresh Token 로직이 없다. JWT 24시간 만료 후 재로그인 필요.

**제안**: `POST /api/auth/refresh` 엔드포인트 추가, Refresh Token은 SESSIONS KV에 30일 TTL로 저장.

---

### 🟡 SEC-05: 그룹 메시지/피드백 Rate Limiting 미적용

**현황**: `POST /api/groups/:id/messages`와 `POST /api/groups/:id/meetings/:mid/feedbacks`에 Rate Limiting이 없어, 스팸/flood 공격에 노출.

**제안**:
```typescript
// groups.ts — 메시지 전송에 Rate Limiting 적용
groupsRouter.post('/:id/messages', authMiddleware, rateLimit({ limit: 30, windowMs: 60_000 }), ...)
```

---

### 🟡 SEC-06: 공유 보고서 수신자 이메일 열거 공격

**현황**: `POST /api/share/report`에서 `recipient_email`로 사용자 존재 여부를 확인하며, 존재하지 않으면 404를 반환한다. 이를 반복하면 등록된 이메일 목록을 열거할 수 있다.

**제안**:
- 존재 여부와 관계없이 동일한 성공 응답 반환 (실제 존재 시에만 DB 저장)
- 또는 해당 엔드포인트에 Rate Limiting 강화 (분당 5회)

---

### 🟢 SEC-07: reading_sessions 중복 INSERT 방지

**현황**: Idempotency key 없이 POST 시 중복 세션 삽입 가능.

**제안**: `book_id + session_date` 복합 UNIQUE 제약 추가 또는 클라이언트에서 idempotency header 전송.

## 2. 리팩토링 제안

### 🔴 REF-01: GroupDetailView 컴포넌트 분리

**현황**: `GroupDetailView.tsx`가 ~350줄의 단일 파일에 ChatTab, MeetingsTab, MeetingCard, FeedbackSection, MembersTab 5개 하위 컴포넌트를 모두 포함하고 있다. 유지보수성과 코드 리뷰 효율이 낮다.

**제안**:
```
src/app/components/groups/
  GroupDetailView.tsx        → 컨테이너 (탭 전환 + 헤더)
  ChatTab.tsx               → 채팅 UI + 메시지 입력
  MeetingsTab.tsx           → 일정 목록 + 생성 폼
  MeetingCard.tsx           → 개별 일정 카드
  FeedbackSection.tsx       → 피드백 목록 + 작성 폼
  MembersTab.tsx            → 멤버 목록 + 역할 관리
```

---

### 🔴 REF-02: removeMember mutation 연결 TODO 해소

**현황**: MembersTab에 멤버 추방 버튼(UserMinus)이 있으나, 실제 `DELETE /api/groups/:id/members/:userId` 호출이 TODO 주석으로 남아있다.

**제안**: `useGroups.ts`에 `useRemoveMember` mutation 추가 후 MembersTab에 연결.

```typescript
// src/hooks/useGroups.ts
export function useRemoveMember() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: ({ groupId, userId }: { groupId: string; userId: string }) =>
      groupsApi.removeMember(groupId, userId),
    onSuccess: (_, { groupId }) => {
      qc.invalidateQueries({ queryKey: queryKeys.groups.detail(groupId) });
    },
  });
}
```

---

### 🟡 REF-03: 공유 보고서 전용 UI 페이지 미구현

**현황**: `shareApi` (5개 메서드) + `useShareInbox/useShareSent/useShareUnreadCount/useShareReport/useMarkReportRead` 훅이 모두 존재하지만, 이를 사용하는 전용 UI 페이지가 없다. 사용자가 받은 보고서를 확인하거나 보낼 수 있는 진입점이 없다.

**제안**: `SharePage.tsx` 생성 — Inbox/Sent 탭 + 보고서 카드 뷰 + 보낼 사람 검색/선택 UI.

```
src/app/pages/SharePage.tsx
  ├── InboxTab: 받은 보고서 카드 리스트 (안읽음 배지)
  ├── SentTab: 보낸 보고서 카드 리스트
  └── ShareModal: 이메일 입력 + 메시지 + 공유 버튼
```

---

### 🟡 REF-04: API 응답 타입 정규화 불일치

**현황**: 기존 엔드포인트는 `{ data: T }` 래퍼를 사용하지만, 일부 groups 엔드포인트는 `{ data: result.results }` 또는 `{ data: { id, ...} }` 등 일관성이 부족하다.

**제안**: 모든 API 응답을 `{ data: T, meta?: { total, page } }` 형태로 통일. 에러는 `{ error: string, code?: string }`.

---

### 🟡 REF-05: 그룹 메시지 cursor 페이지네이션 프론트엔드 미활용

**현황**: 백엔드 `GET /api/groups/:id/messages`에 cursor 기반 페이지네이션이 구현되어 있지만 (`?cursor=&limit=`), 프론트엔드 `useGroupMessages`에서는 limit 없이 전체 조회만 한다.

**제안**: `useInfiniteQuery`로 전환하여 무한 스크롤 채팅 구현.

```typescript
export function useGroupMessages(groupId: string) {
  return useInfiniteQuery({
    queryKey: queryKeys.groups.messages(groupId),
    queryFn: ({ pageParam }) => groupsApi.getMessages(groupId, pageParam, 50),
    getNextPageParam: (lastPage) => lastPage.nextCursor,
    refetchInterval: 10_000,
  });
}
```

---

### 🟢 REF-06: 모임 일정 D1 인덱스 추가

**현황**: `group_meetings` 테이블에 `group_id` 기반 인덱스가 없어, 모임 수가 증가하면 일정 조회 성능이 저하된다.

**제안**:
```sql
CREATE INDEX idx_group_meetings_group ON group_meetings(group_id, meeting_date DESC);
CREATE INDEX idx_group_messages_group ON group_messages(group_id, created_at DESC);
CREATE INDEX idx_meeting_feedbacks_meeting ON meeting_feedbacks(meeting_id);
```

## 3. 성능 최적화

### 🔴 PERF-01: 채팅 폴링 → WebSocket 전환 (Durable Objects)

**현황**: `useGroupMessages`가 10초 폴링으로 D1 쿼리를 반복한다. 동시 접속 N명 × 10초 = 분당 6N회 쿼리.

**영향**: 50명 그룹이 5개 활성화되면 분당 1,500회 D1 쿼리 발생.

**제안**: Cloudflare Durable Objects + WebSocket으로 전환.
- 각 그룹마다 하나의 Durable Object 인스턴스
- WebSocket 연결로 실시간 메시지 브로드캐스트
- D1은 메시지 영속화에만 사용 (write-through)

```typescript
// worker/durable-objects/ChatRoom.ts
export class ChatRoom implements DurableObject {
  sessions: Set<WebSocket> = new Set();
  
  async fetch(request: Request) {
    const pair = new WebSocketPair();
    this.sessions.add(pair[1]);
    // broadcast to all sessions on message
    return new Response(null, { status: 101, webSocket: pair[0] });
  }
}
```

---

### 🟡 PERF-02: GroupsPage 초기 로드 최적화

**현황**: `useGroups()`가 모든 공개 그룹 + 내 그룹을 한 번에 조회한다, 그룹 수 증가 시 페이로드가 커진다.

**제안**:
- 내 그룹: 제한 없이 전체 조회 (소량)
- 공개 그룹: 페이지네이션 적용 (limit=20, offset 기반)
- 검색 시 서버 사이드 필터링 (`?search=키워드`)

---

### 🟡 PERF-03: 공유 보고서 JSON 크기 제한

**현황**: `shared_reports.report_data`에 통계 데이터 전체를 JSON TEXT로 저장한다. 장르별 상세, 월별 데이터 등이 포함되면 수 KB까지 커질 수 있다.

**제안**: 스냅샷 JSON에 핵심 요약만 포함 (상위 5개 장르, 최근 6개월 통계 등으로 제한).

---

### 🟢 PERF-04: GroupsPage 번들 분리

**현황**: GroupsPage가 lazy loaded 되지만, GroupDetailView 내부의 framer-motion + lucide icons가 함께 번들링된다.

**제안**: GroupDetailView를 별도 lazy chunk로 분리.

```typescript
const LazyGroupDetailView = React.lazy(() =>
  import('../components/groups/GroupDetailView').then(m => ({ default: m.GroupDetailView }))
);
```

## 4. UX 개선

### 🟡 UX-01: 공유 보고서 알림 통합

**현황**: `useShareUnreadCount`(30초 폴링)로 안읽은 보고서 수를 추적하지만, 이를 표시하는 UI가 없다.

**제안**:
- SideNav "독서 모임" 항목에 안읽은 보고서 수 배지 표시
- BottomNavBar에도 알림 도트 추가
- 향후 Web Push 연동

---

### 🟡 UX-02: 모임장 위임 기능

**현황**: leader가 활동을 중단하면 그룹이 정체된다. leader는 탈퇴할 수 없고 삭제만 가능.

**제안**:
```
PATCH /api/groups/:id/transfer-leader
body: { newLeaderId: string }
```
- 기존 leader → member 변경, 새 leader → leader 역할 변경
- D1 batch로 원자적 처리

---

### 🟡 UX-03: 채팅 타임스탬프 그룹핑

**현황**: 모든 메시지에 개별 시간(HH:mm)이 표시된다. 연속 메시지가 많으면 시간 표시가 시각적으로 과도해진다.

**제안**:
- 같은 사용자의 연속 메시지는 마지막 메시지에만 시간 표시
- 날짜 변경 시 "─── 2026년 7월 17일 ───" 구분선 삽입

---

### 🟢 UX-04: 빈 그룹 온보딩

**현황**: 처음 "독서 모임" 페이지 진입 시 빈 목록만 표시된다.

**제안**: 비어있을 때 일러스트 + "첫 독서 모임을 만들어보세요!" CTA 강조.
